## 0. 도커 이미지가 저장되는 방식
- 도커 이미지가 빌드될 때 `Dockerfile`에 정의된 명령문을 순서대로 실행하면서 <b>레이어(Layer)</b>가 만들어진다.
<br><br><br>
## 1. 레이어
- 도커 이미지를 `pull` 받게 되면 여러 개로 분리된 듯한 조각(데이터)
- 레이어들은 각각 독립적으로 저장되며 읽기 전용이기 때문에 임의로 수정할 수 없다.
- 실제 레이어에 대해 확인하려면 `docker image inspect` 명령어를 통해 해쉬 값을 얻어, 이 값을 활용할 수 있다.<br><br>
- 모든 명령문이 레이어가 되는 것은 아니며, `RUN`, `ADD`, `COPY` 이 3가지 단계만 레이어로 저장된다.
- 만약 `Dockerfile` 에 위의 세가지 명령문이 수정되면 기존 캐시가 무효가 되어 새로운 레이어를 생성하게 된다.
- `CMD`, `LABEL`, `ENV`, `EXPOSE` 등 메타정보를 다루는 부분은 임시 레이어로 생성되지만 저장되진 않아 <u>도커이미지 사이즈에 영향을 주지 않는다.</u>
<br><br><br>
## 2. 도커 이미지 작성 요령

### (1) 작은 베이스 이미지 선택
- `name:version` : 가장 기본이 되는 이미지, 이미지 배포 및 베이스 이미지용
- 태그를 추가하면, 이미지가 어떤 릴리즈 기반으로 배포되었는지 알 수 있다. (패키지 추가 설치에 용이하다.)<br><br>
> Debian 계열 이미지 권장
- `buster`, `slim-buster`, `stretch`
- https://pythonspeed.com/articles/base-image-python-docker-images/<br><br>
> Alpine 리눅스 도커 이미지
- `name:version-alpine`
- 가볍지만, Python 어플리케이션 환경에서는 추천하지 않는다.
  ```
  - Apline : C 라이브러리로 musl 채택, 유닉스 도구들의 집합인 busybox 기반
  - PyPI에 올라간 파이썬 라이브러리들은 보통 Wheel 포맷을 사용하지만,
  - Alpine은 Wheel을 지원하지 않아 소스코드(.tar.gz)를 내려받아 직접 컴파일 해야하는 경우가 종종 있다.
    (모든 Python 패키지에서 C코드를 컴파일 해야하므로 도커 이미지를 빌드하는 시간이 오래 걸린다.)
    
  * Wheel 포맷은 Python 에서 설치 없이 바로 실행할 수 있다.
  ```
- 참고 : https://pythonspeed.com/articles/alpine-docker-python/ 
<br>

> 그 외 태그
- `slim` : 파이썬을 실행하는 데 필요한 최소한의 패키지만 설치되어 있다.
  - 배포 환경에 용량제한이 심하거나 순수 파이썬만 실행하는 것이 아니라면 권장하지 않는다.
- `windowsservercore` : 윈도우즈 서버 이미지를 기반으로 배포되었다.
<br><br><br>
### (2) 이미지 레이어 개수 줄이기
- 실제 도커 이미지/컨테이너 성능에 영향을 주지는 않지만, 최적화 측면(Dockerfile 가독성과 유지 보수)에서 효율적이다.
- 여러 개로 분리된 명령을 체이닝으로 엮어보자.
- 설치할 패키지 순서를 알파벳 순서로 정렬하면 가독성이 좋고, 중복을 방지할 수 있다.
<br><br><br>
### (3) 애플리케이션 코드를 복사하는 부분은 아래로
- 자칫 의존성 패키지 버전을 `latest` 나 `*`로 해뒀다면 예기치 못하게 패키지 버전이 올라갈 수 있다.
- 코드 복사 명령은 자주 변경되지 않는 명령문 다음에 오는 것이 이미지 빌드 시간을 단축하는데 유리하다.
<br><br><br>
### (4) 프로그래밍 언어마다 패키지 매니저가 제공하는 Lock 파일 시스템을 활용
- Locking 시스템을 갖춘 `Pipenv`나 `Poetry`를 사용하는 것을 추천한다.
- 캐시 레이어의 장점을 얻을 수 있고, 예상치 못한 패키지 버전 업데이트도 방지할 수 있다.<br><br>
- `requirements.txt` 는 `PIP`에서 사용하는 관행적으로 지칭하는 파일이다.
- Python 개발 환경을 기준으로 봤을 때 `PIP` 는 패키지별 상호 의존성에 관계를 관리할 때 부족한 면이 있다.
<br><br><br>
### (5) 멀티-스테이지 빌드 활용
- `Dockerfile` 1개에 `FROM` 구문을 여러 개 두는 방식이다.  
- 각 `FROM` 명령문을 기준으로 스테이지를 구분한다.
- `COPY --from=builder` 를 통해 전 단계 스테이지 빌드에서 생성된 특정 결과물만 새로운 BASE 이미지로 복사해서 이미지를 생성한다.
- 예시
  ```
  Pipfile 에 uwsgi 라는 패키지의 의존성을 명시하고 있고, 이 패키지를 설치하기 위해서는 gcc 가 필요하다.
  하지만 gcc 가 실제 어플리케이션 실행환경에는 필요하지 않다.
  ```
