## QueryEngine.ts: 세션과 턴의 상위 관리자

### 0. Summary

`query.ts`가 단일 루프(질문→답변)를 수행한다면, <br />
`QueryEngine.ts`는 여러 번의 턴이 반복되는 **세션 전체의 상태와 비용**을 책임지는 컨트롤타워이다.

```ts
class QueryEngine {
  mutableMessages: Message[]; // 전체 대화 히스토리 관리
  totalUsage: Usage; // 누적 토큰 사용량 계산 (비용 추적)
  readFileState: FileStateCache; // 중복 파일 읽기 방지 (성능 최적화)
  discoveredSkillNames: Set<string>; // 매 턴 발견된 스킬 초기화
}
```

<br /><br />

### 1. 설계의 핵심: 비대칭 데이터 저장 전략

에이전트가 대화 기록을 저장할 때, 모든 데이터를 똑같이 취급하지 않는다. <br />
데이터의 **복구 필수성**에 따라 동기와 비동기 방식을 영리하게 나누어 사용한다.

<br /><br />

**사용자 메시지: "반드시 적어야 다음 단계로!" (Blocking)** <br />

```ts
// 사용자 메시지: 블로킹 저장 (--resume 복원에 필수)
await recordTranscript(userMessage);
```

만약 저장이 완료되기 전에 프로세스가 종료되면 대화의 맥락이 소실된다. <br />
--resume 기능으로 세션을 이어나가기 위해 `await`를 사용하여 데이터 무결성을 보장한다.

<br />

**어시스턴트 메시지: "일단 말하고, 기록은 나중에!" (Fire-and-forget)** <br />

```ts
// 어시스턴트 메시지: 실행만 하고 완료를 기다리지 않는 비동기 패턴
recordTranscript(assistantMessage); // await 없음
```

답변은 이미 생성되었고, 설령 기록에 실패하더라도 사용자 입력값만 있다면 재생성이 가능하다.<br />
디스크 I/O 대기 시간을 절반 가까이 줄여, 사용자에게 훨씬 빠른 피드백을 제공한다.

<br /><br />

### 2. 성능과 안전의 균형

이러한 비대칭 설계는 에이전틱 루프에서 발생할 수 있는 병목 현상을 우아하게 해결한다.

- 데이터 보호: 잃어버리면 안 되는 데이터(User Input)는 await로 확실히 지킨다.
- 대기 시간 최소화: 기록이 실패하더라도 루프가 멈추지 않게 하여 사용자 경험을 극대화한다.

<br />

### 3. 상태 관리 변수 분석

- `mutableMessages`: AI의 '단기 기억'. 이전 맥락을 기억해 대화를 이어가게 함
- `totalUsage`: accumulateUsage 함수를 통해 세션 내 누적 토큰 소모량 추적
- `readFileState`: I/O 최적화. 동일 파일에 대한 반복적 읽기 접근 제한
- `discoveredSkillNames`: 런타임 최적화. 활성화된 도구/스킬셋 상태 관리

<br />

**Insight:**<br />
좋은 에이전트 설계는 단순히 AI 답변을 잘 받아오는 것이 아니라,<br />
대화의 상태를 어떻게 관리하고 기록 시점을 어떻게 최적화하느냐에 달려 있다.
