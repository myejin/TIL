## 메시지 압축: 컨텍스트 윈도우와의 정교한 전쟁

### 0. Summary

LLM의 컨텍스트 윈도우(Context Window)는 물리적 한계가 존재하며, `413 Prompt Too Long` 에러를 반환한다. <br />
이를 해결하기 위해 **비용** 과 **정보 손실** 사이의 최적점을 찾는 4단계 계층적 압축 전략(4-Layer Compaction Hierarchy)을 사용한다. <br />

**핵심 원칙:**<br />
"항상 가장 저렴하고 과격한 방식부터, 비싸고 정교한 순서로 적용한다."

<br /><br />

### 1. Layer 1: Snip (단순 절단)

가장 원시적이지만 비용이 들지 않는 확실한 방법<br />

- 방식: 오래된 메시지를 통째로 절단하고 최근 컨텍스트만 유지
- 적용: 주로 사용자가 실시간으로 확인하지 않는 헤드리스(Headless, UI 없이 백그라운드에서 실행되는) 세션에서 사용
- 특징: 비용은 0원이나 정보 손실이 매우 크다
- 구현 디테일:
  - `snipTokensFreed` 를 다음 단계인 Auto-Compact에 전달하여, 현재 남아있는 메시지 외 이미 확보된 토큰량을 인지하도록 설계함

```ts
const snipResult = snipModule!.snipCompactIfNeeded(messagesForQuery);
messagesForQuery = snipResult.messages;
snipTokensFreed = snipResult.tokensFreed;
```

<br /><br />

### 2. Layer 2: Microcompact (선택적 클리어)

불필요한 '디테일'만 골라내어 부피를 줄이는 전략 <br />

- 방식: 대규모 도구 실행 결과(예: `file_read` 본문)를 짧은 플레이스홀더(`[Old tool result content cleared]`)로 교체.

- 핵심 기술: 캐시 편집 블록 피닝(Cache Edit Block Pinning)
  - 문제: 중간 데이터를 삭제하면 **프롬프트 캐시** 가 무효화되어 재계산 비용과 지연 시간이 급증함
  - 해결: 현재 캐시된 영역 내 데이터는 삭제하지 않고 **고정(Pin)** 해두었다가, 캐시 범위를 벗어나는 시점에 안전하게 삭제하여 캐시 히트율을 보호함

```ts
const pendingCacheEdits = feature("CACHED_MICROCOMPACT")
  ? microcompactResult.compactionInfo?.pendingCacheEdits
  : undefined;
```

<br /><br />

### 3. Layer 3: Context Collapse (단계적 축소)

Git의 커밋 모델과 유사한 읽기 시점 프로젝션(Read-time Projection) 방식 <br />

- 방식: 원본 메시지 배열은 유지하되, API 요청 시점에만 가공된 요약본을 View 형태로 덮어씌움
- 의도: 기억을 요약하면서도 동시에 성능(캐시)을 챙기기 위함
- 프로세스:
  - Preview: 축소 가능한 블록(예: 연속 파일 읽기) 식별
  - Commit: 축소 결과를 별도 저장소(`collapse`)에 보관하고 `projectView()`가 매 진입 시 리플레이
- 핵심 설계 이점:
  - 데이터 가용성: 원본 데이터(`REPL` 히스토리)를 보존하므로 필요 시 압축을 되돌리거나 다른 형태의 뷰를 생성할 수 있는 유연성을 가짐
  - 캐시 결정론(Determinism): Auto-Compact 와는 달리, 한 번 확정된 축소는 매 턴마다 토큰 하나 틀리지 않고 동일한 시퀀스를 생성함. 덕분에 압축 시점에 캐시가 한 번 리셋된 이후에는 새 접두어가 안정적으로 유지되어 캐시 효율이 극대화됨
  - 시스템 정합성: `notifyCompaction()`을 호출하여 캐시 감지 시스템의 베이스라인을 리셋, 압축된 시점 이후부터 다시 최적의 캐싱 상태를 추적함

<br /><br />

### 4. Layer 4: Auto-Compact (지능형 요약)

최후의 수단으로, AI 모델을 추가 호출하여 전체 맥락 요약 <br />

- 임계값 설정: 컨텍스트 윈도우에서 13K 토큰 버퍼(시스템 프롬프트 및 응답 공간)를 제외한 지점에서 발동
- 안전장치 서킷 브레이커(Circuit Breaker)
  - 요약 요청 자체가 실패하여 발생하는 무한 루프를 방지한다. 3회 연속 실패 시 압축을 포기하고 에러 복구 경로로 제어를 이관한다.
- Auto-Compact 전처리: 요약 성공률을 높이기 위해 요약 대상 자체를 먼저 정리한다.
  - 이미지 스트리핑: 수천 토큰의 이미지와 문서를 플레이스홀더로 대체
  - API 라운드 그룹핑: tool_use(도구 호출 요청)과 tool_result(실행 결과) 쌍을 묶어 데이터 유효성 유지
  - PTL(Prompt-Too-Long) 재시도: 전처리 후에도 한도를 넘을 경우 가장 오래된 그룹부터 순차적/비율제로 드롭

<br /><br />

### 5. 토큰 경고 상태 시스템

사용자에게 시각적 피드백을 제공하기 위한 5단계 상태 머신: <br />

- Normal: 잔여 공간 20K 초과
- Warning: 잔여 20K 미만, "대화가 길어지고 있음" 알림
- Error: 잔여 13K 미만, "곧 자동 압축 발동" 예고
- AutoCompact: 임계값 도달, 자동 압축 수행 및 사용자 공지
- BlockingLimit: 잔여 3K 미만, 수동 압축(`/compact`)만 가능한 최후의 상태.

<br /><br />

**Insight:** <br />

이 설계의 우수성은 하나의 완벽한 전략을 고집하지 않고 비용 최소화와 정보 보존 최대화 사이의 트레이드오프를 인정했다는 점이다. <br />경제학의 파레토 프론티어 원칙처럼, 상황에 따라 최적의 전략을 순차적으로 적용하는 계층적 구조가 에이전트 설계의 핵심이다.
